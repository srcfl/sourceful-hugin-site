Self-hosting + ops

For people running the API and broker themselves — paranoid enterprise, air-gapped lab, or just don't want to depend on Sourceful's uptime. Everything except the AI inference is self-hostable; you bring your own Anthropic key.

Architecture recap

Four moving parts:

  • workbench — static HTML served by anything (Cloudflare Workers, S3, nginx). Repo: sourceful-hugin-site.
  • hugin-api — Go binary. Catalog + AI proxy + GitHub OAuth + NATS signer. Repo: sourceful-hugin.
  • hugin-nats — a NATS server with auth-callout pointed at hugin-api. Self-hosted; any binary release of nats-server works.
  • hugin-agent — the local helper your end users install. Repo: hugin-agent.

Running hugin-api

Single Go binary. Build with make build, image with Dockerfile.api. Runs HTTP on HUGIN_API_PORT (default 8080).

Required environment

VariableWhy
ANTHROPIC_API_KEYYour Anthropic key. Used by the /v1/ai/messages proxy. Without it the AI endpoint returns 503.
GITHUB_OAUTH_CLIENT_IDFrom a GitHub OAuth App. Drives the /v1/auth/login + /v1/auth/callback flow.
GITHUB_OAUTH_CLIENT_SECRETMatching secret.
DRIVERS_PATHPath to the hugin-drivers checkout. Defaults to /app/drivers in the Docker image.
HUGIN_SHARE_HMAC_SECRETServer-side HMAC key for the share-token feature. 32+ random bytes. Without it /v1/share/* returns 503.

Optional — NATS signer

Without these set, the workbench works fine in local-pair (HTTP) mode. With them set, you also get remote-pair mode.

VariableWhy
NATS_URLwss://nats.your-host. The URL the agent should dial.
NATS_OPERATOR_NKEY_SEEDThe signing nkey seed (SOAA…). hugin-api uses it to mint user JWTs.
NATS_ACCOUNT_JWTThe pre-signed account JWT all minted users belong to.
NATS_CALLOUT_USER_CREDSPath to the creds-file for the auth-callout service account. With this set, hugin-api subscribes to the callout subject at boot.
NATS_CALLOUT_SUBJECTOverride the callout subject. Default $SYS.REQ.USER.AUTH.

Running hugin-nats

Standard NATS server with auth-callout configured to delegate to hugin-api. Minimal config: 

port: 4222
http_port: 8222
websocket {
  port: 443
  tls {
    cert_file: "/etc/ssl/cert.pem"
    key_file:  "/etc/ssl/key.pem"
  }
}

# Operator setup — generate once with `nsc`, push to a resolver.
operator: /etc/nats/operator.jwt
system_account: "ACCSYS…"
resolver: { type: full, dir: "/etc/nats/jwt" }

# Auth-callout delegates every CONNECT to hugin-api. The service
# account itself authenticates with a creds file; user JWTs come
# back signed under the HUGIN account.
authorization {
  auth_callout {
    issuer: "AHUGIN…"         # account public key signing user JWTs
    account: HUGIN_CALLOUT    # account the callout service lives in
    users:   ["UCALLOUT…"]    # the callout service user nkey
  }
}

Bootstrap nkeys with nsc. Generate one HUGIN account for end users, one HUGIN_CALLOUT account for the auth-callout service, and a single user in each. Hand the HUGIN_CALLOUT user's creds file to hugin-api as NATS_CALLOUT_USER_CREDS; hand the HUGIN account's signing-nkey seed as NATS_OPERATOR_NKEY_SEED.

Running hugin-agent

Two modes; they can run side by side.

HTTP mode (local-pair)

hugin-agent
# → opens browser to /settings.html#agent_url=…&token=…
Flag / envDefault
--host / HUGIN_AGENT_HOST127.0.0.1. Bind only to localhost unless you trust your LAN.
--port / HUGIN_AGENT_PORT19090.
--token / HUGIN_AGENT_TOKENRandom if unset. Printed on startup + carried in the pairing deep-link fragment.
--pairing-urlhttps://hugin.sourceful-labs.net/settings.html. Override for self-hosted workbenches.
--no-browserDon't auto-open. For headless devices.

NATS mode (remote-pair)

# One-shot: register with your hugin-api and write creds to disk.
hugin-agent --register --gh-token=<your_gh_token> --api-url=https://api.example

# Subsequent runs: agent reads creds.json and dials NATS.
hugin-agent

Creds land at $XDG_CONFIG_HOME/hugin-agent/creds.json (mode 0600). Override with --creds or HUGIN_AGENT_CREDS. JWTs expire after 30 days; the agent prints a renewal hint when within 7 days of expiry.

Security model

What the agent can do on your machine

  • Listen on a TCP port you specified (default 127.0.0.1:19090).
  • Open outbound TCP for Modbus / HTTP / MQTT to addresses the workbench requests.
  • Open outbound WSS to your hugin-api + NATS broker.
  • Execute Lua code the workbench sends, inside a gopher-lua sandbox with no os.*, no io.*, no require.

What the agent can't do

  • Write to disk (creds.json is the only file written; you can audit the path).
  • Auto-update. New versions arrive only through brew upgrade / scoop update / a manual binary swap.
  • Run shell commands. The Lua sandbox doesn't expose os.execute.
  • Phone home. There is no telemetry endpoint. All outbound traffic is to your configured workbench / API / broker.

Token model

  • Local-pair token — random hex, kept only in the agent's memory + your browser's localStorage. Never sent anywhere else.
  • NATS user JWT — signed by your hugin-api, valid 30 days. Scoped to agent.<your_agent_id>.> — can't publish or subscribe outside that prefix.
  • Share token — HMAC-signed by hugin-api, short TTL (default 24h), scoped to read+test on the shared agent only. Revocable by clearing HUGIN_SHARE_HMAC_SECRET (nuclear option; rotates all shares).
  • GitHub OAuth token — held in browser localStorage, scope public_repo. Revoke at github.com/settings/applications.

Catalog hosting

hugin-drivers is a plain Git repo. hugin-api clones it at build time (Dockerfile copies it into the image) and serves it via /v1/catalog. To update the catalog:

  1. Merge a driver PR on srcfl/hugin-drivers (or your fork).
  2. Rebuild + redeploy the hugin-api image with the new commit.

A future iteration will hot-pull instead of build-time-baking, but for beta this is the simplest path.

Observability

hugin-api emits structured logs:

  • Request logs — method, path, status, latency. Standard access log.
  • usage.ai {…} — one JSON line per /v1/ai/messages call. User login, token counts, status, response size, timestamp.
  • nats: … — auth-callout connection state.

Scrape with Better Stack, Grafana Loki, or whatever your stack prefers. No dedicated metrics endpoint yet; track the usage.ai lines to size your AI quota.

Limits + known gaps

  • Agent registry is in-memory — a hugin-api restart loses the GitHub-user-id → agent-id mapping. Already-registered agents stay valid for their JWT lifetime but new pairings have to be re-issued. File-backed persistence is on the roadmap.
  • No metrics endpoint (Prometheus / OpenTelemetry). Logs only.
  • No driver hot-reload — catalog is baked into the image at build time.
  • No multi-broker NATS routing — single broker per deployment.
