Authoring drivers

A Hugin driver is a single Lua file that exposes a DRIVER metadata table plus four lifecycle functions. The runtime calls them in a sandbox; you talk to hardware through a host.* API. ~100 lines for a working meter driver.

The shape of a driver

DRIVER = {
  id                  = "manufacturer_model",
  name                = "Manufacturer Model",
  manufacturer        = "Manufacturer",
  version             = "0.1.0",
  protocols           = { "modbus" },
  capabilities        = { "meter" },
  tested_models       = {},
  verification_status = "experimental",
  description         = "One-sentence purpose.",
  connection_defaults = { port = 502, slave_id = 1 },
}

PROTOCOL = "modbus"

function driver_init(config)
  host.set_make("Manufacturer")
  host.set_sn(config.serial or "unknown")
end

function driver_poll()
  local power = host.modbus_read(0x4000, 2, "i32_be")
  if power then host.emit("meter", { power_w = -power }) end
  return 5000   -- next poll in ms
end

function driver_command(action, power_w, cmd) end
function driver_default_mode() end
function driver_cleanup() end

DRIVER metadata

The metadata table is consumed by the catalog API + by any EMS reading the driver. Every field below is required unless marked otherwise.

FieldTypeNotes
idstringUnique slug. Lowercase + underscores. Convention: manufacturer_model.
namestringHuman-readable display name.
manufacturerstringVendor name as commonly written.
versionstringSemVer. Bump on breaking driver behaviour.
protocolsstring[]One or more of "modbus", "mqtt", "http".
capabilitiesstring[]One or more of "meter", "pv", "battery", "ev".
tested_modelsstring[]Exact model numbers verified on live hardware. Empty = experimental.
verification_statusstring"experimental" · "community-verified" · "vendor-verified".
descriptionstringOne sentence. Appears in the catalog listing.
connection_defaultstable{ port, slave_id } for Modbus; whatever's relevant for HTTP / MQTT.

Lifecycle functions

The runtime calls these in order on every poll cycle.

FunctionCalled whenWhat it does
driver_init(config)Once, before the first pollSet make/serial, parse static config. config is the table from the connection form (host, port, slave_id, plus anything custom).
driver_poll()Every cycleRead registers, call host.emit(...) with the parsed values. Return the next-poll delay in milliseconds.
driver_command(action, power_w, cmd)When an EMS issues a commandBattery / EV chargers only. action ∈ "charge", "discharge", "idle". Most read-only drivers leave this empty.
driver_default_mode()When the EMS resets the deviceReturn the device to a safe default (idle, max-rate, etc.).
driver_cleanup()Once, before the driver unloadsClose subscriptions, save state. Usually empty.

Sign convention — don't break this

Positive watts = power flowing into the site at the grid meter boundary. The driver is the only layer that flips signs.

  • Grid meter: import = +, export = −.
  • PV inverter: generation is always − (it leaves the site at the boundary, so it offsets imports).
  • Battery: charging = + (consumes site energy), discharging = −.
  • EV charger: drawing power = +.

If a vendor's register documents "power generated" as positive, that's a flip your driver must do. host.emit("pv", { power_w = -raw }) is the typical pattern.

Host API reference

Driver metadata

FunctionPurpose
host.set_make(s)Record the manufacturer string. Usually called in driver_init.
host.set_sn(s)Record the device serial number.
host.set_poll_interval(ms)Override the poll cadence. Alternative to returning the delay from driver_poll.

Emitting values

FunctionPurpose
host.emit(channel, table)Publish a measurement. channel ∈ "meter", "pv", "battery", "ev". The table's keys are channel-specific (power_w, soc, etc.).
host.emit_metric(name, number)Publish a free-form gauge or counter. Renders in the Live panel's Metrics block.
host.log(level, msg)Append a log line. level ∈ "debug", "info", "warn", "error".

Modbus

FunctionPurpose
host.modbus_read(addr, count, kind)Read count holding-registers from addr and decode as kind ∈ u16 · i16 · u32_be · u32_le · i32_be · i32_le · f32_be · f32_le. Returns nil on failure.
host.modbus_write(addr, value)Single-register write (FC06).
host.modbus_write_multi(addr, values)Multi-register write (FC16). values is an array of u16.

MQTT

FunctionPurpose
host.mqtt_sub(topic)Subscribe to a topic. Messages buffer until you read them.
host.mqtt_messages()Drain the inbound buffer. Returns an array of {topic, payload}.
host.mqtt_pub(topic, payload)Publish.

HTTP

FunctionPurpose
host.http_get(url, headers)GET request. Returns {status, body}.
host.http_post(url, body, headers)POST request.

Decoders + utilities

FunctionPurpose
host.millis()Monotonic milliseconds since the driver loaded. For rate-limiting and watchdogs.
host.json_encode(t)Lua table → JSON string.
host.json_decode(s)JSON string → Lua table.
host.decode_u32_be / _le, _i32_be / _le, _i16Manual byte-pair decoders if you read raw registers with modbus_read(..., "u16") and want to combine words yourself.

Common pitfalls

  • Word order — most inverters use big-endian (i32_be), but cheap clones often flip. If your reading is wildly wrong, try _le first.
  • Scaling — vendors often store "watts × 10" or "volts × 100" so they can use unsigned integers. Read the manual carefully; the driver should divide by the scale factor before emitting.
  • Sign flip — if your meter shows the right magnitude but wrong direction, you're missing a negation. The Live panel + the catalog API both follow the same convention; if your emission disagrees with the rest of the catalog, your driver is wrong.
  • Polling cadence — most Modbus devices choke at <200ms intervals. Return at least 1000 from driver_poll unless you've verified the device handles faster.
  • No state across polls — Lua locals reset on every driver_poll call. If you need state (rolling averages, last-seen counters), use module-level globals declared outside the function.

Sandbox limits

Drivers run in a gopher-lua sandbox with no os.*, no io.*, no require. The only outside-world access is through host.*. A driver that loops forever in driver_poll is killed after the test duration (5–120s, set by the slider in the workbench). In production an EMS enforces its own timeout — usually 1–5s per poll.
